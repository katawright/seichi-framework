# ADR-001: Front Matter Is Reserved for Programmatic Consumers

**Status:** Accepted

**Date:** 2026-03-23

---

## Decision

YAML front matter is added to framework files only when the metadata is consumed
programmatically — by schema validation, agent routing, or tooling that parses
the fields to make decisions. Files where front matter would serve only as
descriptive labels (topics, audience, scope) do not get front matter.

## Context

The framework originally added front matter to both stage READMEs and guide
files. Stage README front matter carries structural data — inputs, outputs,
checkpoints, execution patterns — that agents and validation tooling act on.
Guide front matter carried three descriptive fields (`id`, `type`, `concerns`)
that were never referenced programmatically:

- `id` was not used by any cross-reference; all links used file paths
- `type` was always `guide`; two other enum values were never used
- `concerns` was a lossy topic index that the guide headings communicated better

We evaluated whether richer guide front matter (audience, prerequisites, stage
relevance, full topic tags) would help agents orient. The conclusion was that
any metadata specific enough to aid filtering risks causing agents to skip files
that contain useful contextual information for non-primary audiences. Guides are
prose — agents should evaluate them by reading the overview and headings, not by
parsing metadata.

## Consequences

- Guide files (`guides/*.md`) have no front matter. `guide.schema.json` and
  `agentic-workflow.schema.json` were removed.
- Stage READMEs retain front matter because their metadata is structural — they
  are the canonical source for all stage metadata including agent-facing fields
  (autonomy, oversight, RACI, revisit conditions).
- Before adding front matter to any file type, contributors should identify the
  programmatic consumer. If the only consumer is "a human or agent reading the
  file," the information belongs in the document body, not front matter.

## Alternatives considered

- **Keep guide front matter as-is:** Low maintenance cost but establishes a
  precedent that all files should have front matter, leading to metadata that no
  tooling consumes.
- **Enrich guide front matter with audience, prerequisites, and stage
  relevance:** Evaluated and rejected. Richer metadata improves filtering but
  creates a risk that agents skip files containing useful secondary information.
  The filtering benefit does not outweigh the missed-context cost.
