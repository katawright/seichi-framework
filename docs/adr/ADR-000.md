# ADR-000: Use Architecture Decision Records

**Status:** Accepted

**Date:** 2026-03-23

---

## Decision

Use Architecture Decision Records (ADRs) to document significant framework
design decisions, following Michael Nygard's lightweight ADR format. ADRs live
in `docs/adr/` within the framework repository.

## Context

This framework has been under development since approximately v0.2.0 with many
design decisions made along the way — checkpoint taxonomy, front matter
conventions, stage modeling, schema design — none of which were formally
recorded. The rationale behind these decisions exists only in git history and
contributor memory, making it difficult for future contributors (including the
original author) to understand why things are the way they are.

ADRs are being adopted late. Many decisions that warranted an ADR were made
before this point. We will not attempt to backfill all past decisions, but will
document them going forward and retroactively when a past decision is revisited
or questioned.

## What qualifies as an ADR

An ADR records a decision that meets one or more of these criteria:

- **Hard to reverse** — the decision shapes conventions, file structure, or
  processes that many files or contributors depend on
- **Constrains future choices** — establishes a principle or precedent that
  future decisions should follow
- **Would surprise a future contributor** — the reasoning isn't obvious from the
  code or file structure alone

ADRs are **not** for implementation details like renaming a variable,
refactoring a schema key, or fixing a bug. Those belong in commit messages. If a
change can be understood entirely from the diff and a one-line commit message,
it doesn't need an ADR.

## Immutability

ADRs are immutable once accepted. Only the **Status** field may change. If a
decision is revisited, write a new ADR that supersedes the old one — do not edit
the original. This preserves the reasoning chain: future readers can see what
was decided, why, and what later changed.

## Statuses

- **Draft** — decision is being explored; the ADR may still change before
  acceptance
- **Accepted** — decision is in effect
- **Superseded by ADR-NNN** — a later ADR replaces this decision
- **Supersedes ADR-NNN** — this ADR replaces an earlier decision

When superseding, update the old ADR's status to `Superseded by ADR-NNN` and add
`Supersedes ADR-NNN` below the status in the new ADR. This creates bidirectional
references so readers can follow the chain in either direction.

## Process

- **Numbering:** Sequential (`ADR-000`, `ADR-001`, ...).
- **One ADR per decision:** Each ADR captures a single architectural decision.
  If a decision area requires multiple decisions, write multiple ADRs.
- **Location:** `docs/adr/` in the framework repository. These are decisions
  about building the framework, not framework guidance for users — they should
  not be confused with stage artifacts or guides.
- **Audience:** Framework contributors and maintainers, including AI agents
  working on the framework.

## Alternatives considered

- **Meta repository only:** Keeps ADRs private but inaccessible to external
  contributors and AI agents working in the framework repo.
- **Mutable ADRs with dated sections:** Group related decisions in a single ADR
  and append new sections over time. Rejected because it obscures the decision
  history — readers can't tell which parts were original and which were added
  later, and it encourages recording implementation details alongside
  architectural decisions.
