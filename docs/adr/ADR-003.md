# ADR-003: Tier-Aware Inline Annotations for Framework Templates

**Status:** Accepted

**Date:** 2026-03-27

---

## Decision

Add tier-aware inline HTML comment annotations to template sections that are
inapplicable or should be simplified at Minimal tier, using the
`<!-- Minimal: ... | Standard: ... | Enterprise: ... -->` convention already
partially established in the templates.

## Context

Simulation run-1 (2026-03-27, DataDigest CLI scenario) produced 26 framework
feedback items. Four high-priority items — FB-01, FB-14, FB-17, and FB-22 — all
share the same root cause: agents wrote N/A placeholder entries in template
sections that were inapplicable for their project tier.

The agents correctly understood what Minimal tier required. The templates did
not. Several sections in the templates already use
`<!-- Minimal: ... | Standard: ... -->` annotations, and those sections worked
correctly — agents adapted their content to the right depth. The problem was
inconsistent application: sections without annotations defaulted to full
Standard-tier treatment.

Three approaches were evaluated:

1. **Tier-specific template files** — separate templates per tier (e.g.,
   `initiation-brief-minimal.md`, `initiation-brief-standard.md`). Rejected:
   approximately 3× the file count for equivalent content. Tiers differ in
   content depth, not structure — a structural divergence does not exist to
   justify separate files. Maintenance overhead grows proportionally with every
   template change.

2. **Machine-readable tier markers** — structured comments like
   `<!-- TIER:standard+ -->...<!-- /TIER -->` that tooling can strip to produce
   clean per-tier templates. Deferred: the SaaS tooling (Theia) that would
   consume these markers does not yet require them, and adding the syntax before
   a consumer exists adds complexity without benefit. The prose annotation
   convention does not prevent this upgrade.

3. **Prose inline annotations** — extends the existing `<!-- Minimal: ... -->`
   convention that already works in practice, requires no tooling, and directly
   addresses agent behavior at template-read time. Chosen.

A complementary design decision: a blockquote convention note is added to each
template (similar to the existing `[ASSUMED]` convention block), making the
annotation semantics self-contained for agents filling in the template.

## Consequences

- N/A ceremony from FB-01, FB-14, and FB-17 is resolved. Agents receive explicit
  skip or simplify instructions rather than inferring behavior from absent
  guidance.
- FB-22 (Increment Design / Implementation overlap) is partially addressed via
  cross-reference annotations. A full structural merge of those two stages is
  explicitly out of scope.
- The annotation convention is now consistent across all templates. Future
  template authors should follow the convention documented in STYLE_GUIDE.md
  (Template file conventions → Template annotation convention).
- Machine-readable tier markers remain a viable future upgrade if Theia requires
  programmatic template generation.

Two related changes were made alongside the annotations:

- **Pre-Mortem no longer marked optional.** The simulation's pre-mortem
  predicted the primary adoption risk that materialized in UAT. The "(Optional)"
  label was removed from the Initiation Brief heading and a Minimal-tier
  annotation requiring 1–2 bullets minimum was added, ensuring agents do not
  skip the section entirely.
- **Stage Skip Decisions section added to System Design Brief.** The simulation
  noted there was no dedicated template section for recording stage skip
  decisions — the Architect had to find an ad hoc home. A structured table now
  provides a formal skip record referenced by Gate 2.

## Alternatives considered

- **Tier-specific template files:** Cleaner fill-out experience per tier but 3×
  maintenance surface. Tiers differ in content depth only, not structure.
  Rejected.
- **Machine-readable tier markers (`<!-- TIER:standard+ -->`):**
  Forward-compatible with SaaS tooling but adds complexity before a consumer
  exists. Deferred, not rejected — the prose convention does not block this
  upgrade.
